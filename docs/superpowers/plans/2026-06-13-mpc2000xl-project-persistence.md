# MPC2000XL Project Persistence Foundation

## Behavior Slice

This slice adds deterministic, metadata-only project snapshots for the full MPC2000XL clone direction. It is a behavior slice toward the full app, not an MVP and not a claim of exact native MPC2000XL disk or project-file compatibility.

Implemented behavior:

- `mpc_core` exposes a typed `ProjectSnapshot` model with `version = 1`.
- Snapshots include sequence index/name, tempo, selected track, bar count, recorded events, playhead ticks, and playhead tick remainder.
- Snapshots include current program index/name, selected program pad, current pad bank, and pad assignments.
- Snapshots include rights-safe synthetic sample metadata: sample ids and names already generated by the app.
- Recorded sequence events preserve optional sample playback metadata captured at record time.
- `MpcCore::export_project_snapshot` returns a deterministic typed snapshot with pad assignments sorted by pad.
- `MpcCore::restore_project_snapshot` validates the snapshot, restores state, and refreshes LCD.
- `MpcCore::to_project_json`, `MpcCore::from_project_json`, and `MpcCore::restore_project_json` provide JSON helpers with typed errors.
- Restore intentionally leaves transport stopped and disarmed because transport persistence is not safely modeled in this slice.
- Conformance fixtures can optionally export/restore a project snapshot and then assert restored state after optional post-restore events.
- Desktop exposes in-memory `Save Snapshot` and `Load Last Snapshot` controls that retain only the latest JSON string in app state and display snapshot status, size, and version.

## Snapshot Schema Boundary

The snapshot is a repo-owned JSON metadata contract, not a native MPC2000XL file-format implementation.

Persisted:

- Snapshot kind, version, and rights boundary marker.
- UI and machine metadata: mode, selected main field, current pad bank, selected program pad, event count, last playback metadata, playhead ticks, and playhead remainder.
- Sequence metadata: index, name, tempo, selected track, bar count, and recorded events.
- Program metadata: index, name, pad assignments, generated synthetic sample ids/names, level, and pan.

Not persisted:

- Audio bytes, sample file contents, recordings, copied assets, firmware bytes, manuals, service scans, or screenshots.
- Native MPC2000XL disk structures or proprietary binary project data.
- Filesystem paths.
- Transport playing/recording state. Restore always returns stopped/disarmed.

## Validation

Restore rejects invalid metadata instead of silently clamping:

- Unsupported snapshot versions.
- Unexpected snapshot kind or rights boundary.
- Out-of-range sequence index, selected track, tempo, bar count, pad numbers, velocities, program index, level, pan, or playhead remainder.
- Empty sequence, program, sample id, or sample name fields.
- Duplicate pad assignment collisions.
- Recorded events whose playback metadata does not match the recorded track, pad, and velocity.
- Recorded event ticks beyond the restored playhead.

The current range choices are internal guardrails matching the foundation model. They can be revised once exact source evidence is mapped.

## Source And Evidence Status

This slice is backed by repo-owned internal specs, unit tests, and conformance fixtures. Exact owner-manual evidence for project save/load UX, firmware traces, storage naming, native file formats, and invalid-file behavior is still pending.

No proprietary samples, firmware, manuals, service scans, screenshots, binary project files, or copied media are added. Behavior-matrix entries for project persistence are marked as internal-spec/manual-investigation pending exact MPC2000XL storage and file-format evidence.

- `./scripts/verify.sh`

Focused checks added or extended:

- Project snapshot round-trips after main-screen edits and sequence recording.
- Unsupported snapshot version is rejected.
- Invalid assignment pad is rejected.
- Duplicate assignment pads are rejected.
- Restore refreshes LCD.
- Restored projects can still emit sample playback intents.
- Conformance fixture records an assigned pad, exports/restores the snapshot, then confirms recorded metadata and post-restore playback intent survive.

## Next Boundaries

Next persistence slices should map exact source evidence before claiming native MPC2000XL project-file accuracy. Practical next boundaries are named project slots, path-safe file save/load UX, explicit schema migration tests, multi-program snapshots, sequence playback from restored events, native disk-format research, and import/export adapters that keep proprietary or user-owned assets out of git.
