# MPC2000XL Project File Storage Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add rights-safe host-side storage for repo-owned MPC2000XL clone project JSON snapshots.

**Architecture:** Keep deterministic project serialization and validation in `mpc_core`, and isolate host filesystem I/O in `mpc_storage`. The desktop shell uses a text path field plus storage calls, then restores loaded snapshots through the existing core validation API.

**Tech Stack:** Rust workspace crates, `mpc_core` project snapshot APIs, standard-library filesystem I/O, and `eframe`/`egui` desktop controls.

---

## Behavior Slice

This slice adds host-side project file storage for the existing metadata-only project snapshot contract. It is foundation storage for the full app, not a native MPC2000XL disk or project-file compatibility claim.

Implemented behavior:

- `mpc_storage` is a new workspace crate responsible for filesystem-backed project file save/load.
- Project files use the explicit `.mpc2000xl-project.json` suffix.
- The default desktop path is `local-assets/projects/last.mpc2000xl-project.json`; `local-assets/` is ignored by git.
- Save serializes the current `MpcCore` with `MpcCore::to_project_json`.
- Load reads JSON with `mpc_storage`, validates it with `MpcCore::from_project_json`, and returns a typed `ProjectSnapshot`.
- Desktop `Load File` restores the loaded snapshot through `MpcCore::restore_project_snapshot`, preserving the existing stopped/disarmed restore behavior.
- Existing in-memory `Save Snapshot` and `Load Last Snapshot` controls remain available.
- Save and load status includes snapshot version and byte count.

## Rights-Safe File Boundary

Persisted:

- Repo-owned JSON metadata generated by the core project snapshot API.
- Snapshot kind, version, and `metadata_only_no_audio_bytes` rights boundary marker.
- Machine, sequence, recorded-event, program, pad assignment, and synthetic sample metadata already represented by `ProjectSnapshot`.

Not persisted:

- Audio bytes, recordings, sample file contents, proprietary Akai assets, manuals, service scans, firmware bytes, screenshots, or binary fixtures.
- Native MPC2000XL disk structures or proprietary binary project data.
- A claim that `.mpc2000xl-project.json` is readable by real MPC2000XL hardware or software.

## Storage API

The `mpc_storage` crate exposes:

- `PROJECT_FILE_SUFFIX` as `.mpc2000xl-project.json`.
- `DEFAULT_PROJECT_FILE_PATH` as `local-assets/projects/last.mpc2000xl-project.json`.
- `default_project_file_path()` for callers that want a `PathBuf`.
- `save_project_file(core, path) -> Result<ProjectFileReport, ProjectStorageError>`.
- `load_project_file(path) -> Result<ProjectSnapshot, ProjectStorageError>`.
- `load_project_file_with_report(path) -> Result<ProjectFileLoad, ProjectStorageError>` for UI status.
- `ProjectFileReport` with resolved path, byte count, and snapshot version.
- `ProjectStorageError` variants for empty paths, directory paths, invalid suffixes, I/O failures, and core project JSON validation failures.

## Error Handling

Save and load reject empty paths, existing directory paths, and paths without the required suffix. Save performs path validation before serializing or creating parent directories.

Save creates parent directories when needed, writes to a sibling temp file, syncs the temp file, renames it to the final project path, and removes the temp file on failures where possible.

Malformed JSON and invalid project JSON are mapped to `ProjectStorageError::ProjectJson` with the original `ProjectSnapshotError` preserved for display and tests.

Focused checks added:

- Conformance fixture coverage saves to a nested temp `.mpc2000xl-project.json` path, loads through `mpc_storage`, restores into a fresh core, and verifies the metadata-only JSON boundary.
- Wrong suffixes are rejected before parent directories are created.
- Malformed JSON maps to a structured project JSON error.
- Invalid project JSON maps to a structured project JSON validation error.
- Parent directories are created on save.
- Saved JSON contains `metadata_only_no_audio_bytes` and no audio byte or sample file content fields.
- Existing directory paths and empty paths are rejected for load and save.
